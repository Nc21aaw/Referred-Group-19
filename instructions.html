YAML

YAML (a recursive acronym for “YAML Ain’t Markup Language”) is a data serialization language used extensively to specify system configurations. It is equivalent to JSON (JavaScript Object Notation) but has a much more human-friendly syntax.

For a nice presentation of the features and syntax of YAML, see YAML tutorial: Get started in 5 minutes by Educative.

For this assignment, there are five things to keep in mind:

  1. YAML “documents” are made up of one or more fields, each comprising a key (name) and a value.

  2. Keys are always followed by a colon (‘:’). There must be no space between the key and the colon, and there must be at least one space after the colon.

    key: value

  3. Values are just strings. They need not be enclosed in quotation marks (contrast with JSON) unless they contain a colon (‘:’), in which case the entire value must be quoted.

    key: "This value has a colon: it must be quoted."

  4. Multi-line strings are possible, and can be specified by putting a ‘>’ or ‘|’ character on the same line as the key, and the lines in the string after the key, indented by at least one space:

    key: >
 First line of a multi-line string.
 The second line.
 Third line.

 First line of a new paragraph, indicated by the preceding blank
 line.
 Second line of the new paragraph.
 Note that all lines are indented at least one space.

  5. YAML is NOT free-form text. YAML has a strict syntax that must be followed if the document is to be valid.

We are using YAML in an unusual way, to create a text document for human consumption. This is not the main use case for YAML, but it is well suited to this task.

Why can’t we use Word?

If we’re creating a document for human readers, why aren’t we using a tool specifically created for that purpose?

There are several reasons:

We need to gain experience with system configuration languages. Word is very hard for machines to read (have you ever seen a system configuration file in .docx or .doc format?).

Conversely, Word documents are very easy for machines to create, if you provide them with the right input. There are numerous utilities that take (semi-)structured text as input and produce .docx as output. Two that I use extensively:

  • Pandoc, which converts documents in markdown format to a wide variety of different output formats (these instructions were written in markdown, then converted by Pandoc to HTML, PDF, and plain text).
  • docx-templates, a JavaScript library for the node.js platform that consumes JSON input and populates a Word document containing placeholders for values extracted from the JSON input using their keys.

So, we can create Word documents, PDF documents, HTML documents, and plain text documents from the same YAML input file.

In any case, you already know how to use Word, so it’s time to learn something new.

Analyze the five forces

  1. Assess the rivalry between existing competitors to your chosen product or service.

    Write your assessment as a multi-line value for the competitor_rivalry field. Remember that multi-line strings start on the line after the key, and each line must be indented at least one space.

  2. Assess the threat of new entrants into the market for your chosen product or service.

    Write your assessment as a multi-line value for the new_entrant_threat field.

  3. Assess the threat of substitute products or services for your chosen product/service.

    Write your assessment as a multi-line value for the substitute_threat field.

  4. Assess the bargaining power of customers of your product or service.

    Write your assessment as a multi-line value for the customer_power field.

  5. Assess the bargaining power of suppliers to your product or service.

    Write your assessment as a multi-line value for the supplier_power field.

  6. Determine the competitive strategy to pursue for your product or service, chosen from Porter’s four strategies:

    1. Overall cost leadership.
    2. Differentiation.
    3. Focus or niche.
    4. “Stuck in the middle.”

    Write your determination, along with the reasoning behind it, as a multi-line value for the competitive_strategy field.

How to commit and push

The Git commit, pull, and push subcommands are how you “submit” your assignments. These are covered in the Essential Git tutorial under the “Coursework” unit on Canvas.

Here is the sequence of commands you should execute:

  1. Add files. When you create or edit a file, you need to add it to the next change set. You do this using git add:

     git add five_forces.yml

    Then, type:

    git status

    to see what you have added.

  2. Commit your changes. Adding a file to the next change set is the first step toward creating a new change set. Once you have added all of the files you have changed, you need to “commit” those changes to create a new change set:

     git commit

    This will open the text editor that you specified when you configured the core.editor property as part of the app-password-lab. Note: you need to configure core.editor on every workstation that you use, including your own laptop1 (it is only necessary to do this once on your own computer). Now type:

     git status

    This should show that your workspace is “clean,” as in nothing to commit. If not, examine the output from git commit for clues.

  3. Pull your teammates’ changes. While you were working on your own tasks, your teammates are (we hope) working on their tasks. If they commit and push before you do, you will have to “pull” their changes into your own local workspace:

     git pull

    Pay attention to any error messages when you do this! Occasionally, git will report a merge conflict which means you need to edit the files with conflicts to fix them, then add and commit these edits.

  4. If your pull succeeded without error, now you can push your changes to Bitbucket:

     git push

    Again, pay attention to any error messages. Rarely, someone else in your group will push before you have a chance to push; if so, just pull again, resolve any conflicts, then push as above.

Now you are finished: your changes, and those of your teammates, are on Bitbucket where I can pull them after the deadline.

  1. To verify that you have, in fact, successfully pushed your changes, have someone else clone your repository, then type:

     git log

    They should see all of the changes that everyone made. If they don’t, one or more of your commits failed, or someone forgot to push.

You can repeat the above sequence of steps as many times as you want to. This is a good idea, in case something goes wrong: you can easily roll-back to the last commit, so if you commit often, rolling back will not result in much lost work.
